适配器模式

定义

Convert the interface of a class into another interface clients expect.Adapter lets classes work together that couldn’t otherwise because of incompatible interfaces.

将一个接口转换成客户希望的另一个接口，使接口不兼容的哪些类可以一起工作。

其别名为包装器，适配器模式既可以作为类结构模式，也可以作为对象结构性模式。

三种实现方式：1. 类适配器，2. 对象适配器，3. 接口适配器。这三种，都是针对适配者而言的。

如果是类适配器，则是适配器类继承适配者并实现目标抽象类

如果是对象适配器，则是适配器类中拥有适配者的对象（可以是声明也可以是参数传递），并实现目标抽象类

如果是接口适配器，则是适配器类实现适配者（接口），也称之为缺省适配器

其实涉及到的角色简单描述下：

Target（目标角色）：客户端所需接口，可以是一个抽象类，也可以是一个接口，也可以是具体类，一般来说组合大于继承，所以这里一般定义为一个接口。

Adapter（适配器角色）：客户端真正的使用的就是这个类，适配器模式的核心类，将adaptee转换成target。

Adaptee（源角色）：被适配的类。

我的理解

适配，适配，从名字上来理解，是某个东西（adaptee）通过适配器（adapter）变成了另外一个东西（target）。好比变压器，大部分家用的都是220V，供电局通过变压器（adapter）把高压（adaptee）转成了220V（target）。

应用场景

只需记住一点：需要修改一个已经投产中的接口时，适配器模式才是最适合的。

比如系统扩展了，需要使用一个已有的接口，但这个类又不符合系统的接口，这个时候就需要使用适配器模式。

注意：

在系统设计阶段不要考虑使用适配器模式，适配器模式解决的是系统扩展的时候遇到的问题。

总结

写完了这边文章，发现不用适配器模式，也可以很好的解决，比如：采用注解，根据传入的国家类型，得到不同的handler。这个可以参考文章《如何优雅的实现根据不同协议得到不同的处理器》。

现在在回过头来，看spring MVC这里为什么要用适配器模式，这里这里的逻辑先描述下：

  1. Determine handler for the current request.
  2. Determine handler adapter for the current request.
  3. handlerAdapter.handle

这里在来看下适配器模式的应用场景，需要修改已经投产的接口的时候，使用适配器模式是最恰当的。

这里我们可以假想，一开始这里可能没有那么复杂，得到了handler，可能直接就调用了handler的handle方法，但是随着情况越发复杂， 这里的doDispatch方法可能也会处理其他业务，所以适配器就出现了，为了应对系统的扩展，HandlerAdapter作为target，Object作为adaptee，XXXHandlerAdapter作为adapter。

注意：HandlerAdapter这里有5个实现类，每一个类中对应有不同的supports和handle方法的实现，这里采用了策略模式，根据不同的条件，选择不同的策略。
